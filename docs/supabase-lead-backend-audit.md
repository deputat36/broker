# Аудит Supabase backend для онлайн-заявок

Дата проверки: 13 июля 2026 года.

## Вывод

Рабочий приём заявок уже реализован через Web3Forms по схеме проекта «Новостройки Борисоглебска».

Текущий режим:

```yaml
lead_capture:
  mode: "web3forms"
  endpoint: ""
```

Web3Forms отправляет email-копию, после успеха пользователь переходит на `/spasibo/`, а SMS, MAX, ВКонтакте, Web Share и копирование остаются резервными каналами.

Supabase должен подключаться как второй параллельный канал хранения, событий, rate limit и уведомлений. Legacy-исходники Supabase в текущем виде к форме подключать нельзя.

Факт актуального деплоя Edge Function, применённой миграции и работы уведомлений через GitHub-репозиторий не подтверждён.

## Проверенные файлы

- `supabase/functions/broker-public-lead/index.ts`;
- `supabase/migrations/202607070001_create_broker_leads.sql`;
- `assets/js/online-application.js`;
- `online-zayavka.md`;
- `_config.yml`;
- `docs/lead-endpoint-contract.md`.

## Рабочая схема до подключения Supabase

1. Пользователь заполняет форму.
2. Браузер создаёт request ID, атрибуцию и техническую квалификацию.
3. После подтверждения Web3Forms отправляет email-копию.
4. Пользователь переходит на `/spasibo/`.
5. При ошибке остаются все резервные способы связи.

Supabase не является условием работоспособности формы.

## Критические несовместимости legacy backend

### 1. Разный формат payload

Новая форма формирует контракт `schema_version: 1` с вложенными объектами:

```json
{
  "request_id": "...",
  "client": {
    "name": "...",
    "phone": "...",
    "city": "..."
  },
  "mortgage": {
    "scenario": "...",
    "object_type": "..."
  },
  "tracking": {},
  "qualification": {},
  "consent": true
}
```

Legacy Edge Function ожидает старые плоские поля:

```json
{
  "phone": "...",
  "client_name": "...",
  "mortgage_goal": "...",
  "consent_accepted": true
}
```

При прямом подключении новая заявка не пройдёт проверку телефона и согласия.

### 2. Нет серверной идемпотентности

Клиент создаёт `request_id`, но:

- таблица `broker_leads` не содержит отдельного столбца `request_id`;
- нет уникального индекса;
- Edge Function не проверяет повторную отправку;
- сетевой retry может создать дубль.

### 3. Нет серверного rate limit

Honeypot недостаточен. Сейчас отсутствует подтверждённое ограничение:

- по IP;
- по телефону;
- по `request_id`;
- по временному окну.

Клиентское минимальное время заполнения не является защитой сервера и легко обходится.

### 4. CORS требует пересмотра

В разрешённые origins включены основной домен, `www`, общий технический origin `https://deputat36.github.io` и локальные адреса.

Перед production-запуском нужно разрешить только необходимые origins. Ответ для запрещённого origin не должен подставлять рабочий origin в CORS-заголовок.

### 5. Нет подтверждённой операционной обработки Supabase

Legacy функция сохраняет строку, но репозиторий не подтверждает:

- кто просматривает таблицу;
- есть ли уведомление Татьяне;
- работает ли Telegram или CRM;
- кто имеет доступ к данным;
- какой срок хранения установлен;
- как выполняется удаление по запросу клиента.

Запись без уведомления может остаться незамеченной. Web3Forms решает текущую доставку email, но не заменяет будущую очередь обработки Supabase.

### 6. Ответ не возвращает lead ID

Legacy функция возвращает только:

```json
{
  "ok": true,
  "message": "Заявка принята"
}
```

Для поддержки и устранения дублей нужно возвращать стабильный `lead_id` и признак duplicate.

## Положительные стороны legacy backend

- `service_role` берётся из переменной окружения, а не из публичного сайта;
- RLS включён;
- публичная политика `anon insert` не активирована;
- origin проверяется;
- обрабатывается `OPTIONS`;
- телефон и часть текстовых полей нормализуются;
- присутствует honeypot;
- ошибки не раскрывают service role key.

Эти элементы нужно сохранить.

## Требуемая схема v2

Минимально необходимы поля:

```sql
request_id uuid not null unique,
schema_version integer not null default 1,
form_version text,
preferred_contact text,
object_price_text text,
down_payment_text text,
income_type text,
bank_history text,
page_url text,
form_fill_ms integer,
tracking jsonb not null default '{}'::jsonb,
qualification jsonb not null default '{}'::jsonb,
delivery_status text not null default 'received'
```

Денежные вводные нужно сохранять как текст либо нормализовать отдельно. Значения вроде `маткапитал`, `пока нет` или `4 500 000 ₽` нельзя безусловно преобразовывать в numeric.

Для живой базы потребуется новая миграция, созданная штатной командой Supabase CLI. Изменение старого migration-файла не обновляет применённую базу.

## Требования к Edge Function v2

1. Принимать только `POST` и `OPTIONS`.
2. Принимать `schema_version: 1`.
3. Проверять `client`, `mortgage`, `tracking`, `qualification` и `consent`.
4. Проверять формат и длину `request_id`.
5. Использовать `request_id` как ключ идемпотентности.
6. Ограничивать запросы по IP и телефону.
7. Ограничивать размер JSON.
8. Нормализовать телефон.
9. Сохранять исходные денежные вводные как текст.
10. Возвращать `lead_id`, `duplicate` и управляемый статус.
11. Не возвращать stack trace и внутренние ошибки.
12. Не записывать в лог полную заявку и чувствительные данные.
13. Создавать событие обработки.
14. Отправлять Telegram-уведомление или добавлять заявку в рабочую очередь.
15. Не ломать Web3Forms и fallback при собственной ошибке.

## Целевая hybrid-схема

После подготовки Supabase:

1. Web3Forms продолжает отправлять независимую email-копию.
2. Edge Function параллельно сохраняет заявку в Supabase.
3. Успех хотя бы одного канала позволяет показать подтверждение клиенту.
4. В Supabase создаётся событие `created`.
5. Telegram или рабочая очередь уведомляет ответственного.
6. Дубли определяются по `request_id`.
7. Rate limit применяется на сервере.

Конфигурация после успешного smoke-теста:

```yaml
lead_capture:
  mode: "hybrid"
  endpoint: "https://<project>.supabase.co/functions/v1/<function>"
```

## Безопасная последовательность запуска

1. Подключить фактический Supabase-проект.
2. Создать новую миграцию через Supabase CLI.
3. Обновить и задеплоить Edge Function.
4. Настроить secrets только в Supabase.
5. Проверить CORS и отказ для постороннего origin.
6. Проверить идемпотентность и rate limit.
7. Проверить запись заявки, событие и уведомление.
8. Провести smoke-тест success/error/timeout/duplicate/spam.
9. Обновить политику хранения и удаления данных.
10. Указать HTTPS endpoint.
11. Перевести режим с `web3forms` на `hybrid`.
12. Обновить аудит Pages, сохранив обязательную проверку Web3Forms.

## Решение до следующего этапа

- Web3Forms остаётся рабочим основным каналом.
- `lead_capture.endpoint` остаётся пустым.
- Legacy Edge Function не подключается.
- Supabase развивается в issue №7 как второй канал.
- Реальное поступление тестового письма Web3Forms и адрес получателя требуют ручного smoke-теста после публикации.
