# Аудит Supabase backend для онлайн-заявок

Дата проверки: 13 июля 2026 года.

## Вывод

Исходники Supabase backend присутствуют в репозитории, но подключать их к публичной форме пока нельзя.

`lead_capture.mode` должен оставаться `disabled`, а `lead_capture.endpoint` — пустым.

Факт актуального деплоя Edge Function, применённой миграции и реальной доставки заявок получателю через GitHub-репозиторий не подтверждён.

## Проверенные файлы

- `supabase/functions/broker-public-lead/index.ts`;
- `supabase/migrations/202607070001_create_broker_leads.sql`;
- `assets/js/online-application.js`;
- `online-zayavka.md`;
- `_config.yml`.

## Критические несовместимости

### 1. Разный формат payload

Новая форма формирует контракт `schema_version: 1` с вложенными объектами:

```json
{
  "request_id": "...",
  "client": {
    "name": "...",
    "phone": "...",
    "city": "..."
  },
  "mortgage": {
    "scenario": "...",
    "object_type": "..."
  },
  "consent": true
}
```

Текущая Edge Function ожидает старые плоские поля:

```json
{
  "phone": "...",
  "client_name": "...",
  "mortgage_goal": "...",
  "consent_accepted": true
}
```

При прямом подключении новая заявка не пройдёт проверку телефона и согласия.

### 2. Нет серверной идемпотентности

Клиент создаёт `request_id`, но:

- таблица `broker_leads` не содержит отдельного столбца `request_id`;
- нет уникального индекса;
- Edge Function не проверяет повторную отправку;
- повторный клик или сетевой retry может создать дубль.

### 3. Нет серверного rate limit

Honeypot в Edge Function полезен, но недостаточен. Сейчас отсутствует ограничение запросов:

- по IP;
- по телефону;
- по `request_id`;
- по временному окну.

Клиентское минимальное время заполнения не является защитой сервера и легко обходится.

### 4. CORS требует пересмотра

В разрешённые origins включены:

- основной домен;
- `www`-домен;
- общий технический origin `https://deputat36.github.io`;
- локальные адреса.

Перед production-запуском нужно решить, требуется ли технический origin целиком. Без необходимости лучше разрешить только рабочий домен и конкретные локальные адреса для разработки.

Ответ для запрещённого origin не должен подставлять разрешённый рабочий origin в CORS-заголовок.

### 5. Нет подтверждённого получателя

Текущая функция сохраняет строку в таблицу, но репозиторий не подтверждает:

- кто и где просматривает новые заявки;
- есть ли уведомление Татьяне;
- есть ли Telegram, CRM или email-доставка;
- кто имеет доступ к таблице;
- какой срок хранения установлен;
- как выполняется удаление заявки по запросу клиента.

Запись в таблицу без операционного уведомления может создавать незамеченные заявки.

### 6. Ответ не возвращает lead ID

Функция возвращает только:

```json
{
  "ok": true,
  "message": "Заявка принята"
}
```

Для поддержки и устранения дублей желательно возвращать стабильный `lead_id`.

## Положительные стороны текущего backend

- `service_role` берётся из переменной окружения, а не из публичного сайта;
- RLS включён;
- публичная политика `anon insert` не активирована;
- origin проверяется;
- обрабатывается `OPTIONS`;
- телефон и часть текстовых полей нормализуются;
- присутствует honeypot;
- ошибки не раскрывают service role key.

Эти элементы нужно сохранить при переработке.

## Требуемая схема v2

Минимально необходимы поля:

```sql
request_id uuid not null unique,
schema_version integer not null default 1,
form_version text,
preferred_contact text,
object_price_text text,
down_payment_text text,
income_type text,
bank_history text,
page_url text,
form_fill_ms integer,
delivery_status text not null default 'received'
```

Числовые поля можно добавлять только после отдельной нормализации. Ввод пользователя вроде `маткапитал`, `пока нет` или `4 500 000 ₽` нельзя безусловно преобразовывать в numeric.

Для уже применённой базы потребуется отдельная миграция, созданная штатной командой Supabase CLI. Нельзя считать изменение старого migration-файла обновлением живой базы.

## Требования к Edge Function v2

1. Принимать только `POST` и `OPTIONS`.
2. Принимать `schema_version: 1`.
3. Проверять структуру `client`, `mortgage` и `consent`.
4. Проверять формат и длину `request_id`.
5. Использовать `request_id` как ключ идемпотентности.
6. Ограничивать запросы по IP и телефону.
7. Ограничивать размер JSON до разумного значения.
8. Нормализовать телефон.
9. Хранить исходные денежные вводные как текст либо валидировать их отдельно.
10. Возвращать `lead_id`.
11. Не возвращать stack trace и внутренние ошибки.
12. Не записывать в лог полную заявку и чувствительные данные.
13. Отправлять операционное уведомление ответственному получателю.
14. Сохранять fallback сайта независимо от результата endpoint.

## Безопасная последовательность запуска

1. Выбрать реальный канал получения: таблица + уведомление, Telegram, CRM или email.
2. Подключить Supabase-проект к рабочему процессу и проверить фактическое состояние таблицы и функции.
3. Создать новую миграцию через Supabase CLI.
4. Обновить и задеплоить Edge Function.
5. Настроить secrets только в Supabase.
6. Проверить CORS с рабочего домена и отказ для постороннего origin.
7. Проверить idempotency и rate limit.
8. Проверить реальное уведомление получателя.
9. Обновить политику обработки данных.
10. Обновить `scripts/audit-online-application.py`, сняв блокировку режима `disabled`.
11. Указать HTTPS endpoint в `_config.yml`.
12. Провести smoke-тест с тестовой заявкой и удалить тестовые данные.

## Решение до следующего этапа

Публичная форма остаётся полезной и рабочей без backend:

- формирует структурированное сообщение;
- сохраняет источник страницы;
- предзаполняет сценарий;
- предлагает SMS, MAX, ВКонтакте, копирование и системное меню;
- не теряет данные при ошибке будущего endpoint.

Автоматическую серверную доставку включать нельзя до устранения перечисленных блокеров.
