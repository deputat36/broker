# Подготовка исходников перед Jekyll

## Зачем нужен prebuild

Перед `node --check`, source-аудитами и Jekyll build workflow запускает:

```bash
python3 scripts/prepare-jekyll-source.py --write
```

Подготовка существует для детерминированных преобразований, которые нельзя оставлять на усмотрение шаблонизатора или браузера.

Текущие операции:

1. нормализация известных текстовых полей YAML front matter;
2. подключение оптимизированных WebP-фотографий вместо старого SVG-контейнера;
3. минимизация локальной thank-you сводки и безопасный fragment redirect.

## Fail-closed принцип

Фото и privacy-преобразования используют точные ожидаемые блоки.

Если исходный файл изменился и старый либо уже подготовленный маркер не найден, скрипт завершается ошибкой. Workflow не должен молча публиковать частично подготовленный вариант.

Нельзя заменять этот принцип:

- широкими регулярными выражениями без проверки результата;
- `try/except`, который скрывает несовпадение;
- продолжением сборки после неизвестного состояния;
- runtime-патчем после отправки данных или загрузки аналитики.

## Идемпотентность

После первого прохода source-audit запускает:

```bash
bash scripts/check-jekyll-preparation.sh
```

Скрипт:

1. вычисляет SHA-256 бинарного Git diff после первого прохода;
2. запускает подготовку второй раз;
3. выполняет `git diff --check`;
4. повторно вычисляет SHA-256 diff;
5. требует одинаковые хеши;
6. требует нулевое число изменений для front matter, фото и thank-you privacy.

Таким образом, результат подготовки не зависит от количества запусков.

## Диагностика

Второй проход пишет `prepare-second-pass.log`.

При ошибке source-audit загружает артефакт `source-audit-diagnostics`, содержащий:

- `prepare-second-pass.log`;
- `source-audits.log`, если suite успел запуститься.

## Правила изменения prebuild

При добавлении новой операции необходимо:

- использовать точные входные и выходные маркеры;
- поддержать состояние «уже подготовлено»;
- добиться нулевого изменения на втором проходе;
- добавить отдельный аудит фактического результата;
- проверить подготовленный исходник и итоговый Pages-артефакт;
- описать причину и границы преобразования в документации.

## Долгосрочное направление

Предпочтительно постепенно переносить подготовленный результат в канонические исходники и удалять временные преобразования. До этого момента проверка идемпотентности гарантирует воспроизводимость CI и защищает от накопительных изменений при повторной сборке.
