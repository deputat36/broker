# Подготовка исходников перед Jekyll

## Назначение

Перед source-аудитами и Jekyll build workflow запускает:

```bash
bash scripts/prepare-jekyll-source.sh
```

Ключевая клиентская разметка больше не создаётся скрытой подменой. Privacy-safe JavaScript, WebP-фото, условные ресурсы анкеты и thank-you parser хранятся прямо в канонических исходниках.

## Что делает wrapper

Последовательно выполняются:

1. `audit-canonical-layout.py` — проверяет общий layout;
2. `audit-canonical-prepared-sources.py` — проверяет privacy JS, фото-разметку и связанные audit-маркеры;
3. `prepare-jekyll-source.py` — нормализует только известные текстовые поля YAML front matter.

Если канонический маркер отсутствует или вернулся устаревший вариант, сборка завершается до Jekyll.

## Front matter

Исторические Markdown-файлы могут содержать двоеточие с пробелом внутри некавыченного текстового значения. Psych способен интерпретировать такую строку как структуру и потерять `title`, `description`, `schema` или `permalink`.

Нормализатор заключает известные текстовые значения в JSON-совместимые кавычки. Он не меняет HTML-содержимое страниц.

## Идемпотентность

После первого прохода выполняется:

```bash
bash scripts/check-jekyll-preparation.sh
```

Проверка:

- сравнивает SHA-256 Git diff до и после второго прохода;
- выполняет `git diff --check`;
- требует подтверждения канонического layout;
- требует подтверждения остальных prepared-source;
- требует нулевого изменения front matter на втором проходе.

Повторный запуск не должен менять репозиторий.

## Что удалено из prebuild

Больше не выполняются автоматические подмены:

- SVG-портрета на WebP;
- расширенного localStorage на минимальный `request_id/expires_at`;
- query redirect на fragment redirect;
- глобальных ресурсов анкеты на условные Liquid-блоки;
- старых маркеров в аудитах доставки.

Эти изменения находятся в исходниках и защищены fail-closed аудитами.

## Диагностика

При ошибке source-audit сохраняет:

- `prepare-second-pass.log`;
- `source-audits.log`, если suite успел запуститься.

## Правила дальнейших изменений

Новая runtime-подмена допускается только как временная миграция с отдельным обоснованием. Предпочтительный порядок:

1. записать корректный результат в исходник;
2. добавить source-аудит;
3. проверить Pages-артефакт;
4. не усложнять prebuild преобразованием, которое можно хранить канонически.
