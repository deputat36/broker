# Автоматическая проверка доступности сайта

## Назначение

Сайт должен оставаться пригодным для клавиатурной навигации, экранных дикторов и пользователей с увеличением интерфейса после любых изменений layout, формы, меню или CTA.

Обязательная проверка:

```bash
python3 scripts/audit-accessibility-structure.py _site
```

Она запускается внутри `scripts/post-build-check.sh` для каждого pull request.

## Что проверяется на каждой HTML-странице

- `lang="ru"` у корневого элемента;
- ровно один `<main>`;
- `id="main-content"` и `tabindex="-1"` у main;
- ссылка быстрого перехода на `#main-content`;
- отсутствие повторяющихся `id`;
- существование элементов, на которые ссылаются `aria-controls`, `aria-labelledby` и `aria-describedby`;
- доступное имя у каждой навигационной области;
- ровно один H1 внутри main;
- отсутствие пропущенных уровней заголовков;
- `alt` у каждого изображения;
- положительные `width` и `height` у локальных изображений;
- label, вложенный label либо ARIA-имя у каждого видимого поля формы;
- доступное имя у каждой кнопки и ссылки;
- явный корректный `type` у каждой кнопки;
- `href` у каждой ссылки;
- `rel="noopener"` у ссылок с `target="_blank"`.

## Что аудит не заменяет

Структурная проверка не заменяет:

- ручную навигацию клавишей Tab;
- проверку фокуса после открытия мобильного меню;
- тестирование экранным диктором;
- проверку контрастности в браузерных инструментах;
- проверку при увеличении масштаба;
- реальную мобильную приёмку.

Эти действия остаются в issue ручной приёмки опубликованного сайта.

## Правила изменения

Нельзя отключать проверку ради отдельной страницы. Если появляется исключительный элемент, сначала нужно:

1. убедиться, что он имеет доступное имя;
2. связать управляющий элемент и область через существующий id;
3. добавить label или ARIA-описание полю формы;
4. сохранить логичную иерархию заголовков;
5. проверить клавиатурное поведение вручную.

Новые исключения допустимы только для технических элементов, которые действительно не участвуют в интерфейсе пользователя, и должны быть явно объяснены в pull request.
