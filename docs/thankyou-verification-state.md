# Проверенное состояние страницы благодарности

## Проблема

Страница `/spasibo/` доступна по прямой ссылке. Поэтому сам факт её открытия не подтверждает, что форма была отправлена и хотя бы один канал действительно принял обращение.

## Безопасное поведение

Страница собирается в нейтральном состоянии:

- заголовок сообщает о проверке подтверждения;
- номер обращения равен `—`;
- передача отмечена как неподтверждённая;
- пользователь может вернуться к форме или связаться с Татьяной напрямую.

Успешное состояние включается JavaScript только при наличии корректного request ID:

- UUID;
- резервного номера формата `IP-YYYYMMDD-XXXXXXXX`.

ID читается из очищенного fragment-контекста либо из минимального localStorage со сроком хранения. Сценарий, город, телефон и другие данные на странице благодарности не сохраняются.

## Аналитика

Цель `lead_thankyou_view` отправляется только после проверки request ID.

Чтобы повторное открытие одной и той же страницы в рамках браузерной сессии не увеличивало конверсию, request ID используется только как ключ дедупликации в `sessionStorage`. Он не передаётся в `dataLayer` и цель аналитики.

Прямой визит без подтверждения создаёт только техническое событие `lead_thankyou_unverified_view` и не вызывает conversion goal.

## Автоматическая проверка

```bash
python3 scripts/audit-thankyou-verification-state.py ./_site
```

Аудит требует:

- нейтральное состояние в готовом HTML;
- отсутствие видимого сообщения об успехе до выполнения JavaScript;
- проверку request ID перед сменой состояния;
- дедупликацию подтверждённой цели;
- отсутствие conversion goal в неподтверждённой ветке.

Проверка включена в обязательный post-build suite.
