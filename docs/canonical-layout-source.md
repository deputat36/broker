# Канонический layout

## Назначение

`_layouts/default.html` должен хранить уже подготовленную клиентскую разметку. Сборка не должна зависеть от скрытой подмены layout для ключевых требований безопасности и производительности.

В исходнике обязательно находятся:

- условное подключение `online-application.css` только на `/online-zayavka/`;
- условное подключение `online-application.js` только на странице анкеты;
- адаптивный preload WebP-портрета;
- thank-you parser, который читает только технический request ID из fragment или legacy query;
- очистка query и fragment до аналитики.

## Автоматический контроль

Перед любым prebuild запускается:

```bash
python3 scripts/audit-canonical-layout.py
```

Проверка fail-closed: если в layout вернётся глобальная загрузка ресурсов анкеты, старый preload либо `scenario/status` страницы благодарности, source-audit и production build остановятся.

Существующие подготовители временно остаются как совместимая страховка. Для канонического layout они обязаны давать нулевое изменение, что дополнительно подтверждает idempotence-проверка.

## Границы

В этом этапе не канонизирован длинный `assets/js/online-application.js`: его privacy-преобразование остаётся отдельным техническим долгом. Клиентское поведение формы и каналы доставки заявок не меняются.
