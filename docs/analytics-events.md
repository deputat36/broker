Источники событий: `assets/js/main.js`, `assets/js/application-inputs.js`, `assets/js/application-preparation.js`, `assets/js/online-application.js` и страница `/spasibo/`.

## Текущий канал приёма

Рабочий режим в `_config.yml`:

```yaml
lead_capture:
  mode: "web3forms"
  web3forms_access_key: "UUID"
  web3forms_endpoint: "https://api.web3forms.com/submit"
  endpoint: ""
  thank_you_path: "/spasibo/"
```

Web3Forms отправляет email-копию заявки. Конкретный получатель определяется используемым access key и подтверждается отдельным smoke-тестом.

Дополнительный Supabase/CRM endpoint пока пустой. После подготовки backend v2 режим можно заменить на `hybrid`: Web3Forms останется email-копией, а endpoint станет параллельным хранилищем и каналом уведомлений.

## Короткая, сложная и подробная заявка

Сразу отображаются только обязательные для первичного контакта поля:

- имя;
- телефон;
- город;
- предпочтительный способ связи;
- задача;
- согласие.

Стоимость объекта, первоначальный взнос, доход, банковская история и комментарий находятся в раскрываемом блоке «Добавить подробности».

Если пользователь приходит из одного из 12 сложных региональных маршрутов, дополнительно показывается блок «Что уже успели проверить». В нём можно отметить выполненные проверки и написать, что осталось непонятно. При обычном переходе из меню этот блок скрыт и не усложняет короткую форму.

Событие `online_application_more_open` помогает понять, какой процент пользователей готов заполнить расширенную анкету. События `online_application_complex_prefill` и `online_application_preparation_check` показывают использование подготовительных маршрутов. Сравнивать их следует с `online_application_start`, `online_application_prepare` и `lead_submit`.

Телефон нормализуется только в текущей форме и не сохраняется как черновик. Событие `online_application_phone_error` показывает, насколько часто пользователи не могут пройти проверку номера.

## Атрибуция источника и подготовка

`assets/js/main.js` сохраняет в браузере:

- первое касание;
- последнее касание;
- текущие UTM-метки;
- `gclid`, `yclid`, `ymclid`, `vkclid` и другие рекламные идентификаторы;
- referrer, URL и заголовок страницы.

Срок хранения атрибуции — 90 дней с последнего посещения. Безопасная сводка последней заявки без телефона и полного текста хранится не более 24 часов.

Ссылки с услуг и региональных страниц дополнительно передают `source`. Сложные маршруты также передают `journey=complex` и `stage=route`. Форма использует параметры для предзаполнения сценария, объекта, региона и блока подготовки.

При прямой онлайн-отправке подготовка передаётся отдельным объектом `preparation`, отдельными полями Web3Forms и разделом в `fields_json`. Исходный комментарий клиента не изменяется.

Для резервных каналов формируется безопасная текстовая сводка `ПОДГОТОВКА ДО ОБРАЩЕНИЯ`. Она добавляется к готовой заявке в textarea, SMS, копировании, ВКонтакте, MAX и Web Share, но не записывается в поле комментария формы.

## Ошибки дополнительного endpoint

Сырой `error_code`, correlation request ID и персональные сведения в Метрику не передаются. Ошибки объединяются только в пять стабильных категорий: `validation`, `rate_limit`, `rejected`, `backend` и `request`.

Общая цель `online_application_endpoint_error` показывает число ответов дополнительного endpoint с безопасным error envelope. Категорийные цели нужны для диагностики класса проблемы, но не означают, что вся заявка потеряна.

В режиме `hybrid` Web3Forms и Supabase работают параллельно. Если Web3Forms успешен, а дополнительный endpoint ошибся, сохраняются `online_application_direct_success`, `lead_submit` и переход на `/spasibo/`; endpoint-ошибка учитывается отдельно.

## Состояние hybrid-доставки

После завершения ожидаемых каналов браузер фиксирует только одну техническую категорию: `web3forms_only`, `supabase_only` или `both`.

Эта категория:

- не показывается клиенту;
- не меняет страницу `/spasibo/`;
- не содержит request ID или данные заявки;
- не означает, что оператор уже прочитал обращение;
- используется для контроля расхождений между email и Supabase.

Receipt goals относятся только к best-effort подтверждению `both`. Ошибка квитанции не отменяет `online_application_direct_success` и `lead_submit`.

## Как интерпретировать воронку

`online_application_prepare` означает только подготовку заявки.

`online_application_direct_success` означает успешный технический ответ хотя бы одного канала. В текущем режиме это Web3Forms.

`lead_submit` фиксируется непосредственно перед переходом на страницу благодарности.

`lead_thankyou_view` подтверждает, что пользователь увидел `/spasibo/`, но не подтверждает, что Татьяна уже прочитала письмо или связалась с клиентом.

Для контроля реальных заявок нужно периодически сверять события с входящими письмами. После подключения Supabase или CRM сверка проводится также с таблицей заявок и Telegram-уведомлениями.

## Квалификация заявки

Форма присваивает технический статус:

- `hot` — заполнены основные финансовые параметры или описана сложная ситуация;
- `warm` — задача понятна, но части вводных не хватает;
- `cold` — требуется уточнение данных.

Короткая заявка закономерно чаще получает статус `cold` или `warm`. Это не снижает ценность обращения: статус используется только для порядка обработки и не является оценкой вероятности банковского одобрения.

Отметки подготовки сами по себе не повышают вероятность одобрения и не заменяют проверку брокером. Они нужны только для понимания, с какого этапа начинать разговор.

## Резервные способы

Если Web3Forms недоступен, готовый текст не теряется. Пользователь может:

- отправить SMS;
- открыть ВКонтакте и вставить скопированную заявку;
- скопировать готовую заявку для MAX;
- скопировать текст вручную;
- использовать системное меню «Поделиться»;
- позвонить.

Для сложного маршрута каждый текстовый резервный способ должен содержать раздел `ПОДГОТОВКА ДО ОБРАЩЕНИЯ` ровно один раз.

## Где задаётся ID Метрики

Файл `_config.yml`:

```yaml
yandex_metrika_id: ""
```

Пока значение пустое, код Метрики не выводится.

## Проверка после публикации

1. Отправить короткую тестовую заявку без раскрытия дополнительных полей.
2. Повторить тест с блоком «Добавить подробности».
3. Открыть форму из сложного регионального маршрута и проверить сценарные отметки.
4. Убедиться, что Web3Forms получает отдельные поля подготовки и исходный комментарий без служебной склейки.
5. Проверить, что textarea, SMS, копирование, ВКонтакте, MAX и Web Share содержат сводку подготовки ровно один раз.
6. Проверить форматирование номера, ввод через `8` и `+7`, а также ошибку неполного номера.
7. Убедиться, что письмо поступило получателю Web3Forms.
8. Проверить тему, номер заявки, телефон, сценарий и источник.
9. Проверить переход на `/spasibo/`.
10. Убедиться, что телефон и полный текст подготовки не хранятся в `sterlikovaMortgageLastLead`.
11. Проверить fallback при искусственной ошибке Web3Forms.
12. Проверить пять категорий будущего endpoint, технический номер и отсутствие сырого error code в интерфейсе и Метрике.
13. Проверить hybrid: успешный Web3Forms не должен быть отменён ошибкой дополнительного endpoint.
14. Выполнить `docs/hybrid-delivery-state-smoke.md` для `web3forms_only`, `supabase_only`, `both` и receipt failure.
15. После подключения Метрики проверить endpoint error goals, delivery state goals, `online_application_max`, `lead_submit` и `lead_thankyou_view`.
