# Минимизация данных Web3Forms

## Проблема

Web3Forms email ранее получал одни и те же сведения несколькими способами:

- отдельными понятными полями;
- готовым текстом обращения;
- полной JSON-копией `fields_json`.

`fields_json` повторно содержал имя, телефон, город, ипотечные вводные, технический контекст и квалификацию. Для рабочего email это не давало отдельной пользовательской функции, но увеличивало объём дублируемых персональных данных.

## Канонический email payload

Web3Forms получает:

- имя, телефон, город и предпочтительный способ связи;
- ипотечный сценарий и добровольные финансовые вводные;
- request ID и безопасный технический контекст;
- квалификацию обращения;
- разрешённые UTM и click ID;
- безопасный `tracking_json` без произвольных query-параметров;
- готовый текст обращения;
- для сложного маршрута — отдельные поля подготовки, `preparation_json` и раздел в `message`.

Полный `fields_json` не отправляется.

## Граница с будущим серверным каналом

Минимизация относится к Web3Forms email. Будущий закрытый endpoint получает канонический JSON payload напрямую, потому что серверной обработке необходима структурированная схема, идемпотентность и operational workflow.

Публичный Supabase endpoint остаётся выключенным до внешней приёмки.

## Автоматическая проверка

```bash
python3 scripts/audit-web3forms-data-minimization.py ./_site
```

Аудит проверяет исходный и собранный JavaScript:

- `fields_json` отсутствует;
- отдельные рабочие поля и `message` сохранены;
- safe `tracking_json` сохранён;
- preparation передаётся отдельными полями без повторного полного payload;
- публичная политика и контракт подготовки соответствуют реализации.
