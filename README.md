# Сайт ипотечного брокера Татьяны Стерликовой

SEO- и рекламный сайт для сбора обращений по ипотечным консультациям и сопровождению.

Репозиторий: `deputat36/broker`  
Публичный адрес: `https://sterlikova-ipoteka.ru/`

Технический адрес GitHub Pages остаётся запасным и не используется в рекламе.

## Позиционирование

Сайт объединяет два направления:

1. Частный ипотечный брокер Татьяна Стерликова.
2. Ипотечное сопровождение клиентов компании «ЭТАЖИ» в рамках сделки через компанию.

Первичную онлайн-консультацию можно запросить из любого города России. Возможность дальнейшего сопровождения зависит от региона, банка, объекта и согласованной задачи.

Локальные SEO-разделы подготовлены для:

- Борисоглебска и Борисоглебского городского округа;
- Грибановского района;
- Поворино и Поворинского района.

Локальные страницы не ограничивают дистанционные обращения.

## Рабочий приём заявок

Основной действующий канал — Web3Forms:

- форма отправляет email-копию;
- получатель определяется публичным идентификатором формы;
- после успешного ответа пользователь переходит на `/spasibo/`;
- при ошибке остаются SMS, MAX, ВКонтакте, Web Share и копирование текста.

Текущая публичная конфигурация:

```yaml
lead_capture:
  mode: "web3forms"
  web3forms_access_key: "UUID"
  web3forms_endpoint: "https://api.web3forms.com/submit"
  endpoint: ""
  thank_you_path: "/spasibo/"
  timeout_ms: 8000
  min_fill_ms: 3000
```

Web3Forms access key является публичным идентификатором формы, а не серверным секретом. Фактический получатель проверяется отдельным smoke-тестом в issue №8.

## Онлайн-анкета

Форма версии 2 использует прогрессивное заполнение.

Сразу показываются:

- имя;
- телефон;
- город или населённый пункт;
- предпочтительный способ связи;
- задача клиента;
- согласие на обработку данных.

В блоке «Добавить подробности» доступны:

- тип объекта;
- примерная стоимость;
- первоначальный взнос;
- способ подтверждения дохода;
- история обращений в банки;
- комментарий.

Для сложных сценариев форма дополнительно показывает добровольный блок подготовки. Отметки передаются отдельно от комментария клиента.

Форма не предназначена для паспорта, СНИЛС, банковских реквизитов, кодов подтверждения, фотографий документов или полного кредитного отчёта.

## Телефонное поле

Номер:

- принимает ввод через `+7`, `8` или десять цифр;
- приводится к единому отображению;
- повторно проверяется перед подготовкой заявки;
- не обрезает лишние цифры незаметно;
- не сохраняется в localStorage или sessionStorage.

## Атрибуция

`assets/js/main.js` сохраняет:

- первое рекламное касание;
- последнее касание;
- текущие UTM-метки;
- рекламные click ID;
- URL, заголовок страницы и referrer.

Срок локальной рекламной атрибуции — 90 дней с последнего посещения.

Страницы услуг и региональные посадочные страницы передают параметр `source`. Форма по возможности подставляет сценарий, тип объекта и регион.

## Безопасная локальная сводка

После успешной отправки на 24 часа сохраняется только краткая сводка без телефона и полного текста заявки:

- request ID;
- сценарий;
- тип объекта;
- город;
- техническая квалификация;
- время отправки;
- использованные каналы.

Сводка нужна странице `/spasibo/` и удаляется после истечения срока при следующем открытии.

## Подготовленный Supabase backend

Supabase-контур находится в репозитории, но не развёрнут как рабочий публичный канал.

Канонический порядок содержит десять миграций:

1. базовая таблица заявок;
2. request ID, события и rate limit;
3. структурированный preparation;
4. защищённая сводка уведомления;
5. атомарная доставка;
6. ручной retry и health;
7. retention;
8. индивидуальные privacy-запросы;
9. operational guard;
10. browser-safe статус restricted duplicate.

Единственный актуальный список: `docs/supabase-migration-order.md`.

Backend предусматривает:

- `schema_version: 1`;
- уникальность request ID;
- серверную валидацию и нормализацию телефона;
- точный CORS;
- ограничение размера JSON;
- атомарный rate limit;
- таблицу событий;
- Telegram через серверные secrets;
- ручной retry только для `failed`;
- обезличенный health-контроль;
- retention preview и подтверждённое обезличивание;
- privacy workflow;
- operational guard для ограниченных заявок;
- fail-closed ответы при неполном backend.

## Минимальный публичный ответ Supabase

Новая, обычная повторная и restricted-заявка используют единый success envelope:

```json
{
  "ok": true,
  "success": true,
  "duplicate": false,
  "request_id": "<номер обращения>",
  "notification_status": "disabled"
}
```

Браузеру не возвращаются:

- внутренний `lead_id`;
- CRM status;
- `technical_priority`;
- `qualification`;
- контактные и ипотечные данные;
- tracking и `raw_payload`;
- причины privacy-ограничений.

Внутренний UUID, qualification и технический приоритет сохраняются в защищённой серверной записи.

Контракт: `docs/public-lead-response-contract.md`.

## Конфиденциальность и хранение

Подготовлены отдельные контуры:

- отключённая по умолчанию retention policy;
- индивидуальные запросы на ограничение или обезличивание;
- retention hold;
- запрет автоматической обработки restricted-заявки;
- точечный CRM/export snapshot;
- обезличенный notification health.

Supabase-очистка не удаляет уже доставленные Web3Forms email, SMS, MAX, ВКонтакте, Telegram или сведения сторонней CRM.
